43  /**
44    * A minimal set of BDB DB-JE compatibility constants and static methods, for
45    * internal use only.
46    *
47    * Two versions of this class, with the same public interface but different
48    * implementations, are maintained in parallel in the DB and JE source trees.
49    * By the use of the constants and methods in this class, along with a script
50    * that moves the source code from JE to DB, the source code in certain
51    * packages is kept "portable" and is shared by the two products. The script
52    * translates the package names from com.sleepycat.je to com.sleepycat.db, and
53    * perform other fix-ups as described further below.
54    *
55    * The JE directories that contain portable code are:
56    *
57    * src/com/sleepycat/bind
58    * /collections
59    * /persist
60    * /util
61    * test/com/sleepycat/bind
62    * /collections
63    * /persist
64    * /util
65    *
66    * In DB, these sources are stored in the following locations:
67    *
68    * Sources:
69    * src/java
70    * Tests:
71    * test/java/compat
72    *
73    * To keep this source code portable there are additional coding rules, above
74    * and beyond the standard rules (such as coding style) for all JE code.
75    *
76    * + In general we should try to use the JE/DB public API, since it is usually
77    * the same or similar in both products. If we use internal APIs, they will
78    * always be different and will require special handling.
79    *
80    * + When there are differences between products, the first choice for
81    * handling the difference is to use a DbCompat static method or constant.
82    * This keeps the source code the same for both products (except in this
83    * DbCompat class).
84    *
85    * + When JE-only code is needed -- for example, some APIs only exist in JE,
86    * and special handling of JE exceptions is sometimes needed -- the
87    * following special comment syntax can be used to bracket the JE-only code:
88    *
89    * <!-- begin JE only -->
90    * JE-only code goes here
91    * <!-- end JE only -->
92    *
93    * This syntax must be used inside of a comment: either inside a javadoc
94    * section as shown above, or inside a single-line comment (space before
95    * last slash is to prevent ending this javadoc comment):
96    *
97    * /* <!-- begin JE only --> * /
98    * JE-only code goes here
99    * /* <!-- end JE only --> * /
100    *
101    * All lines between the <!-- begin JE only --> and <!-- end JE only -->
102    * lines, and including these lines, will be removed by the script that
103    * transfers code from JE to DB.
104    *
105    * + When DB-only code is needed, the code will exist in the JE product but
106    * will never be executed. For DB-only APIs, we hide the API from the user
107    * with the @hidden javadoc tag. The @hidden tag is ignored on the DB side.
108    * We do not have a way to remove DB-only code completely from the JE
109    * product, because we do not use a proprocessor for building JE.
110    *
111    * + Because DatabaseException (and all subclasses) are checked exceptions in
112    * DB but runtime exceptions in JE, we cannot omit the 'throws' declaration.
113    * Another difference is that DB normally throws DatabaseException for all
114    * errors, while JE has many specific subclasses for specific errors.
115    * Therefore, any method that calls a DB API method (for example,
116    * Database.get or put) will have a "throws DatabaseException" clause.
117    *
118    * + Special consideration is needed for the @throws clauses in javadoc. We do
119    * want to javadoc the JE-only exceptions that are thrown, so the @throws
120    * for these exceptions should be inside the "begin/end JE only" brackets.
121    * We also need to document the fact that DB may throw DatabaseException for
122    * almost any method, so we do that with a final @throws clause that looks
123    * like this:
124    *
125    * @throws DatabaseException the base class for all BDB exceptions.
126    *
127    * This is a compromise. JE doesn't throw this exception, but we've
128    * described it in a way that still makes some sense for JE, sort of.
129    *
130    * + Other special handling can be implemented in the transfer script, which
131    * uses SED. Entire files can be excluded from the transfer, for example,
132    * the JE-only exception classes. Name changes can also be made using SED,
133    * for example: s/LockConflictException/DeadlockException/. See the
134    * db/dist/s_je2db script for details.
135    */
