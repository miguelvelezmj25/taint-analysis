26  /**
27    * Specifies the attributes of an environment.
28    *
29    *

To change the default settings for a database environment, an application

30    * creates a configuration object, customizes settings and uses it for
31    * environment construction. The set methods of this class validate the
32    * configuration values when the method is invoked. An
33    * IllegalArgumentException is thrown if the value is not valid for that
34    * attribute.

35    *
36    *

Most parameters are described by the parameter name String constants in

37    * this class. These parameters can be specified or individually by calling
38    * {@link #setConfigParam}, through a Properties object passed to {@link
39    * #EnvironmentConfig(Properties)}, or via properties in the je.properties
40    * files located in the environment home directory.

41    *
42    *

For example, an application can change the default btree node size

43    * with:

44    *
45    * 


46    *     envConfig.setConfigParam(EnvironmentConfig.LOCK_TIMEOUT, "250 ms");


47    * 


48    *


49    * 
Some commonly used environment attributes have convenience setter/getter


50    * methods defined in this class.  For example, to change the default


51    * lock timeout setting for an environment, the application can instead do


52    * the following:


53    * 


54    *     // customize an environment configuration


55    *     EnvironmentConfig envConfig = new EnvironmentConfig();


56    *     // will throw if timeout value is invalid


57    *     envConfig.setLockTimeout(250, TimeUnit.MILLISECONDS);


58    *     // Open the environment using this configuration.


59    *     Environment myEnvironment = new Environment(home, envConfig);


60    * 


61    *

        109    *

        288          /**
        

        289            * Configures the number of bytes to be used as a secondary, off-heap cache.
        

        290            *
        

        291            * The off-heap cache is used to hold record data and Btree nodes when
        

        292            * these are evicted from the "main cache" because it overflows. Eviction
        

        293            * occurs according to an LRU algorithm and takes into account the user-
        

        294            * specified {@link CacheMode}. When the off-heap cache overflows, eviction
        

        295            * occurs there also according to the same algorithm.
        

        296            * 
        

        297            * The main cache is in the Java heap and consists primarily of the Java
        

        298            * objects making up the in-memory Btree data structure. Btree objects are
        

        299            * not serialized the main cache, so no object materialization is needed to
        

        300            * access the Btree there. Access to records in the main cache is therefore
        

        301            * very fast, but the main cache has drawbacks as well: 1) The larger the
        

        302            * main cache, the more likely it is to have Java GC performance problems.
        

        303            * 2) When the Java heap exceeds 32GB, the "compressed OOPs" setting no
        

        304            * longer applies and less data will fit in the same amount of memory. For
        

        305            * these reasons, JE applications often configure a heap of 32GB or less,
        

        306            * and a main cache that is significantly less than 32GB, leaving any
        

        307            * additional machine memory for use by the file system cache.
        

        308            * 
        

        309            * The use of the file system cache has performance benefits, but
        

        310            * also has its own drawbacks: 1) There is a significant redundancy
        

        311            * between the main cache and the file system cache because all data and
        

        312            * Btree information that is logged (written) by JE appears in the file
        

        313            * system and may also appear in the main cache. 2) It is not possible
        

        314            * for dirty Btree information to be placed in the file system
        

        315            * cache without logging it, this logging may be otherwise unnecessary, and
        

        316            * the logging creates additional work for the JE cleaner; in other words,
        

        317            * the size of the main cache alone determines the maximum size of the
        

        318            * in-memory "dirty set".
        

        319            * 
        

        320            * The off-heap cache is stored outside the Java heap using a native
        

        321            * platform memory allocator. The current implementation relies on
        

        322            * internals that are specific to the Oracle and IBM JDKs; however, a
        

        323            * memory allocator interface that can be implemented for other situations
        

        324            * is being considered for a future release. Records and Btree objects are
        

        325            * serialized when they are placed in the off-heap cache, and they must be
        

        326            * materialized when they are moved back to the main cache in order to
        

        327            * access them. This serialization and materialization adds some CPU
        

        328            * overhead and thread contention, as compared to accessing data directly
        

        329            * in the main cache. The off-heap cache can contain dirty Btree
        

        330            * information, so it can be used to increase the maximum size of the
        

        331            * in-memory "dirty set".
        

        332            * 
        

        333            * NOTE: If an off-heap cache is configured but cannot be used because
        

        334            * that native allocator is not available in the JDK that is used, an
        

        335            * {@code IllegalStateException} will be thrown by the {@link Environment}
        

        336            * or {@link com.sleepycat.je.rep.ReplicatedEnvironment} constructor. In
        

        337            * the current release, this means that the {@code sun.misc.Unsafe} class
        

        338            * must contain the {@code allocateMemory} method and related methods, as
        

        339            * defined in the Oracle JDK.
        

        340            * 
        

        341            * When configuring an off-heap cache you can think of the performance
        

        342            * trade-offs in two ways. First, if the off-heap cache is considered to be
        

        343            * a replacement for the file system cache, the serialization and
        

        344            * materialization overhead is not increased. In this case, the use of
        

        345            * the off-heap cache is clearly beneficial, and using the off-heap cache
        

        346            * "instead of" the file system cache is normally recommended. Second, the
        

        347            * off-heap cache can be used along with a main cache that is reduced in
        

        348            * size in order to compensate for Java GC problems. In this case, the
        

        349            * trade-off is between the additional serialization, materialization and
        

        350            * contention overheads of the off-heap cache, as compared to the Java GC
        

        351            * overhead.
        

        352            * 
        

        353            * When dividing up available memory for the JVM heap, the off-heap cache,
        

        354            * and for other uses, please be aware that the file system cache and the
        

        355            * off-heap cache are different in one important respect. The file system
        

        356            * cache automatically shrinks when memory is needed by the OS or other
        

        357            * processes, while the off-heap cache does not. Therefore, it is best to
        

        358            * be conservative about leaving memory free for other uses, and it is not
        

        359            * a good idea to size the off-heap cache such that all machine memory will
        

        360            * be allocated. If off-heap allocations or other allocations fail because
        

        361            * there is no available memory, the process is likely to die without any
        

        362            * exception being thrown. In one test on Linux, for example, the process
        

        363            * was killed abruptly by the OS and the only indication of the problem was
        

        364            * the following shown by {@code dmesg}.
        

        365            * 
        

        366            * Out of memory: Kill process 28768 (java) score 974 or sacrifice child
        

        367            * Killed process 28768 (java)
        

        368            *    total-vm:278255336kB, anon-rss:257274420kB, file-rss:0kB
        

        369            *

            1807            * Normally, records (key-value pairs) are stored on disk as individual
            

            1808            * byte sequences called LNs (leaf nodes) and they are accessed via a
