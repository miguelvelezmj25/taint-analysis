63  /**
64    * DbTree represents the database directory for this environment. DbTree is
65    * itself implemented through two databases. The nameDatabase maps
66    * databaseName-> an internal databaseId. The idDatabase maps
67    * databaseId->DatabaseImpl.
68    *
69    * For example, suppose we have two databases, foo and bar. We have the
70    * following structure:
71    *
72    * nameDatabase idDatabase
73    * IN IN
74    * | |
75    * BIN BIN
76    * +-------------+--------+ +---------------+--------+
77    * . | | . | |
78    * NameLNs NameLN NameLN MapLNs for MapLN MapLN
79    * for internal key=bar key=foo internal dbs key=53 key=79
80    * dbs data= data= data= data=
81    * dbId79 dbId53 DatabaseImpl DatabaseImpl
82    * | |
83    * Tree for foo Tree for bar
84    * | |
85    * root IN root IN
86    *
87    * Databases, Cursors, the cleaner, compressor, and other entities have
88    * references to DatabaseImpls. It's important that object identity is properly
89    * maintained, and that all constituents reference the same DatabaseImpl for
90    * the same db, lest they develop disparate views of the in-memory database;
91    * corruption would ensue. To ensure that, all entities must obtain their
92    * DatabaseImpl by going through the idDatabase.
93    *
94    * DDL type operations such as create, rename, remove and truncate get their
95    * transactional semantics by transactionally locking the NameLN appropriately.
96    * A read-lock on the NameLN, called a handle lock, is maintained for all DBs
97    * opened via the public API (openDatabase). This prevents them from being
98    * renamed or removed while open. See HandleLocker for details.
99    *
100    * However, for internal database operations, no handle lock on the NameLN is
101    * acquired and MapLNs are locked with short-lived non-transactional Lockers.
102    * An entity that is trying to get a reference to the DatabaseImpl gets a short
103    * lived read lock just for the fetch of the MapLN, and a DatabaseImpl usage
104    * count is incremented to prevent eviction; see getDb and releaseDb. A write
105    * lock on the MapLN is taken when the database is created, deleted, or when
106    * the MapLN is evicted. (see DatabaseImpl.isInUse())
107    *
108    * The nameDatabase operates pretty much as a regular application database in
109    * terms of eviction and recovery. The idDatabase requires special treatment
110    * for both eviction and recovery.
111    *
112    * The issues around eviction of the idDatabase center on the need to ensure
113    * that there are no other current references to the DatabaseImpl other than
114    * that held by the mapLN. The presence of a current reference would both make
115    * the DatabaseImpl not GC'able, and more importantly, would lead to object
116    * identity confusion later on. For example, if the MapLN is evicted while
117    * there is a current reference to its DatabaseImpl, and then refetched, there
118    * will be two in-memory versions of the DatabaseImpl. Since locks on the
119    * idDatabase are short lived, DatabaseImpl.useCount acts as a reference count
120    * of active current references. DatabaseImpl.useCount must be modified and
121    * read in conjunction with appropriate locking on the MapLN. See
122    * DatabaseImpl.isInUse() for details.
123    *
124    * This reference count checking is only needed when the entire MapLN is
125    * evicted. It's possible to evict only the root IN of the database in
126    * question, since that doesn't interfere with the DatabaseImpl object
127    * identity.
128    *
129    * Another dependency on usage counts was introduced to prevent MapLN deletion
130    * during cleaner and checkpointer operations that are processing entries for a
131    * DB. (Without usage counts, this problem would have occurred even if DB
132    * eviction were never implemented.) When the usage count is non-zero it
133    * prohibits deleteMapLN from running. The deleted state of the MapLN must not
134    * change during a reader operation (operation by a thread that has called
135    * getDb and not yet called releaseDb).
136    *
137    * Why not just hold a MapLN read lock during a reader operation?
138    * --------------------------------------------------------------
139    * Originally this was not done because of cleaner performance. We afraid that * either of the following solutions would not perform well:
140    * + If we get (and release) a MapLN lock for every entry in a log file, this
141    * adds a lot of per-entry overhead.
142    * + If we hold the MapLN read lock for the duration of a log file cleaning
143    * (the assumption is that many entries are for the same DB), then we block
144    * checkpoints during that period, when they call modifyDbRoot.
145    * Therefore, the usage count is incremented once per DB encountered during log
146    * cleaning, and the count is decremented at the end. This caching approach is
147    * also used by the HA replayer. In both cases, we do not want to lock the
148    * MapLN every entry/operation, and we do not want to block checkpoints or
149    * other callers of modifyDbRoot. It is acceptable, however, to block DB
150    * naming operations.
151    *
152    * In addition we allow modifyDbRoot to run when even the usage count is
153    * non-zero, which would not be possible using a read-write locking strategy.
154    * I'm not sure why this was done originally, perhaps to avoid blocking. But
155    * currently, it is necessary to prevent a self-deadlock. All callers of
156    * modifyDbRoot first call getDb, which increments the usage count. So if
157    * modifyDbRoot was to check the usage count and retry if non-zero (like
158    * deleteMapLN), then it would loop forever.
159    *
160    * Why are the retry loops necessary in the DbTree methods?
161    * --------------------------------------------------------
162    * Three methods that access the MapLN perform retries (forever) when there is
163    * a lock conflict: getDb, modifyDbRoot and deleteMapLN. Initially the retry
164    * loops were added to compensate for certain slow operations. To solve that
165    * problem, perhaps there are alternative solutions (increasing the lock
166    * timeout). However, the deleteMapLN retry loop is necessary to avoid
167    * deleting it when the DB is in use by reader operations.
168    *
169    * Tendency to livelock
170    * --------------------
171    * Because MapLN locks are short lived, but a reader operation may hold a
172    * MapLN/DatabaseImpl for a longer period by incrementing the usage count,
173    * there is the possibility of livelock. One strategy for avoiding livelock is
174    * to avoid algorithms where multiple threads continuously call getDb and
175    * releaseDb, since this could prevent completion of deleteMapLN. [#20816]
176    */
