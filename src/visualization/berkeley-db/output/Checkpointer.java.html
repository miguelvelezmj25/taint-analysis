70  /**
71    * The Checkpointer looks through the tree for internal nodes that must be
72    * flushed to the log. Checkpoint flushes must be done in ascending order from
73    * the bottom of the tree up.
74    *
75    * Checkpoint and IN Logging Rules
76    * -------------------------------
77    * The checkpoint must log, and make accessible via non-provisional ancestors,
78    * all INs that are dirty at CkptStart. If we crash and recover from that
79    * CkptStart onward, any IN that became dirty (before the crash) after the
80    * CkptStart must become dirty again as the result of replaying the action that
81    * caused it to originally become dirty.
82    *
83    * Therefore, when an IN is dirtied at some point in the checkpoint interval,
84    * but is not logged by the checkpoint, the log entry representing the action
85    * that dirtied the IN must follow either the CkptStart or the FirstActiveLSN
86    * that is recorded in the CkptEnd entry. The FirstActiveLSN is less than or
87    * equal to the CkptStart LSN. Recovery will process LNs between the
88    * FirstActiveLSN and the end of the log. Other entries are only processed
89    * from the CkptStart forward. And provisional entries are not processed.
90    *
91    * Example: Non-transactional LN logging. We take two actions: 1) log the LN
92    * and then 2) dirty the parent BIN. What if the LN is logged before CkptStart
93    * and the BIN is dirtied after CkptStart? How do we avoid breaking the rules?
94    * The answer is that we log the LN while holding the latch on the parent BIN,
95    * and we don't release the latch until after we dirty the BIN. The
96    * construction of the checkpoint dirty map requires latching the BIN. Since
97    * the LN was logged before CkptStart, the BIN will be dirtied before the
98    * checkpointer latches it during dirty map construction. So the BIN will
99    * always be included in the dirty map and logged by the checkpoint.
100    *
101    * Example: Abort. We take two actions: 1) log the abort and then 2) undo the
102    * changes, which modifies (dirties) the BIN parents of the undone LNs. There
103    * is nothing to prevent logging CkptStart in between these two actions, so how
104    * do we avoid breaking the rules? The answer is that we do not unregister the
105    * transaction until after the undo phase. So although the BINs may be dirtied
106    * by the undo after CkptStart is logged, the FirstActiveLSN will be prior to
107    * CkptStart. Therefore, we will process the Abort and replay the action that
108    * modifies the BINs.
109    *
110    * Exception: Lazy migration. The log cleaner will make an IN dirty without
111    * logging an action that makes it dirty. This is an exception to the general
112    * rule that actions should be logged when they cause dirtiness. The reasons
113    * this is safe are:
114    * 1. The IN contents are not modified, so there is no information lost if the
115    * IN is never logged, or is logged provisionally and no ancestor is logged
116    * non-provisionally.
117    * 2. If the IN is logged non-provisionally, this will have the side effect of
118    * recording the old LSN as being obsolete. However, the general rules for
119    * checkpointing and recovery will ensure that the new version is used in
120    * the Btree. The new version will either be replayed by recovery or
121    * referenced in the active Btree via a non-provisional ancestor.
122    *
123    * Checkpoint Algorithm TODO update this
124    * --------------------
125    * The final checkpointDirtyMap field is used to hold (in addition to the dirty
126    * INs) the state of the checkpoint and highest flush levels. Access to this
127    * object is synchronized so that eviction and checkpointing can access it
128    * concurrently. When a checkpoint is not active, the state is CkptState.NONE
129    * and the dirty map is empty. When a checkpoint runs, we do this:
130    *
131    * 1. Get set of files from cleaner that can be deleted after this checkpoint.
132    * 2. Set checkpointDirtyMap state to DIRTY_MAP_INCOMPLETE, meaning that dirty
133    * map construction is in progress.
134    * 3. Log CkptStart
135    * 4. Construct dirty map, organized by Btree level, from dirty INs in INList.
136    * The highest flush levels are calculated during dirty map construction.
137    * Set checkpointDirtyMap state to DIRTY_MAP_COMPLETE.
138    * 5. Flush INs in dirty map.
139    * + First, flush the bottom two levels a sub-tree at a time, where a
140    * sub-tree is one IN at level two and all its BIN children. Higher
141    * levels (above level two) are logged strictly by level, not using
142    * subtrees.
143    * o If je.checkpointer.highPriority=false, we log one IN at a
144    * time, whether or not the IN is logged as part of a subtree,
145    * and do a Btree search for the parent of each IN.
146    * o If je.checkpointer.highPriority=true, for the bottom two
147    * levels we log each sub-tree in a single call to the
148    * LogManager with the parent IN latched, and we only do one
149    * Btree search for each level two IN. Higher levels are logged
150    * one IN at a time as with highPriority=false.
151    * + The Provisional property is set as follows, depending on the level
152    * of the IN:
153    * o level is max flush level: Provisional.NO
154    * o level is bottom level: Provisional.YES
155    * o Otherwise (middle levels): Provisional.BEFORE_CKPT_END
156    * 6. Flush VLSNIndex cache to make VLSNIndex recoverable.
157    * 7. Flush UtilizationTracker (write FileSummaryLNs) to persist all
158    * tracked obsolete offsets and utilization summary info, to make this info
159    * recoverable.
160    * 8. Log CkptEnd
161    * 9. Delete cleaned files from step 1.
162    * 10. Set checkpointDirtyMap state to NONE.
163    *
164    * Per-DB Highest Flush Level
165    * --------------------------
166    * As mentioned above, when the dirty map is constructed we also determine the
167    * highest flush level for each database. This is the maximum Btree level at
168    * which a dirty node exists in the DB.
169    *
170    * When logging a node below the maxFlushLevel, we add the parent to the dirty
171    * map. It may or may not have been added when the dirty map was constructed.
172    * The idea is to flush all ancestors of all nodes in the dirty map, up to and
173    * including the maxFlushLevel, even if those ancestors were not dirty when the
174    * dirty map was constructed.
175    *
176    * This is done to avoid orphaning a dirty node as shown in this example.
177    *
178    * IN-A (root level=4)
179    * / \
180    * (d) IN-B IN-C (maxFlushLevel=3)
181    * \
182    * (d) IN-D
183    *
184    * IN-C is not dirty (d) when the dirty map is constructed, but it will be
185    * logged because its child (IN-D) is dirty, and it is not above maxFlushLevel.
186    *
187    * If IN-C were not logged, and there were a crash after the checkpoint, the
188    * changes to IN-D would be lost. IN-D would not be replayed by recovery
189    * because it is logged provisionally, and it would not be accessible via its
190    * parent. This is because only nodes at maxFlushLevel are logged
191    * non-provisionally. The actions that led to the changes in IN-D may not be
192    * replayed either, because they may appear before the firstActiveLsn
193    * associated with the checkpoint.
194    *
195    * When log files are to be deleted at the end of the checkpoint (after being
196    * processed by the log cleaner), the maxFlushLevel is increased by one.
197    * This is to ensure that LSNs in deleted files will not be fetched during
198    * recovery. Such files are in the FileSelector.CLEANED state, which means
199    * they have been processed by the cleaner since the last checkpoint.
200    *
201    * TODO: Document circumstances and motivation for the extra flush level.
202    *
203    * Lastly, for Database.sync or a checkpoint with MinimizeRecoveryTime
204    * configured, we will flush all the way to the root rather than using the
205    * maxFlushLevel computed as described above.
206    *
207    * Provisional.BEFORE_CKPT_END
208    * ---------------------------
209    * See Provisional.java for a description of the relationship between the
210    * checkpoint algorithm above and the BEFORE_CKPT_END property.
211    *
212    * Coordination of Eviction and Checkpointing
213    * ------------------------------------------
214    * Eviction can proceed concurrently with all phases of a checkpoint, and
215    * eviction may take place concurrently in multiple threads. This concurrency
216    * is crucial to avoid blocking application threads that perform eviction and
217    * to reduce the amount of eviction required in application threads.
218    *
219    * Eviction calls Checkpointer.coordinateEvictionWithCheckpoint, which calls
220    * DirtyINMap.coordinateEvictionWithCheckpoint, just before logging an IN.
221    * coordinateEvictionWithCheckpoint returns whether the IN should be logged
222    * provisionally (Provisional.YES) or non-provisionally (Provisional.NO).
223    *
224    * Other coordination necessary depends on the state of the checkpoint:
225    * + NONE: No additional action.
226    * o return Provisional.NO
227    * + DIRTY_MAP_INCOMPLETE: The parent IN is added to the dirty map, exactly
228    * as if it were encountered as dirty in the INList during dirty map
229    * construction.
230    * o IN is root: return Provisional.NO
231    * o IN is not root: return Provisional.YES
232    * + DIRTY_MAP_COMPLETE:
233    * o IN level GTE highest flush level: return Provisional.NO
234    * o IN level LT highest flush level: return Provisional.YES
235    *
236    * In general this is designed so that eviction will use the same provisional
237    * value that would be used by the checkpoint, as if the checkpoint itself were
238    * logging the IN. However, there are several conditions where this is not
239    * exactly the case.
240    *
241    * 1. Eviction may log an IN with Provisional.YES when the IN was not dirty at
242    * the time of dirty map creation, if it became dirty afterwards. In this
243    * case, the checkpointer would not have logged the IN at all. This is safe
244    * because the actions that made that IN dirty are logged in the recovery
245    * period.
246    * 2. Eviction may log an IN with Provisional.YES after the checkpoint has
247    * logged it, if it becomes dirty again. In this case the IN is logged
248    * twice, which would not have been done by the checkpoint alone. This is
249    * safe because the actions that made that IN dirty are logged in the
250    * recovery period.
251    * 3. An intermediate level IN (not bottom most and not the highest flush
252    * level) will be logged by the checkpoint with Provisional.BEFORE_CKPT_END
253    * but will be logged by eviction with Provisional.YES. See below for why
254    * this is safe.
255    * 4. Between checkpoint step 8 (log CkptEnd) and 10 (set checkpointDirtyMap
256    * state to NONE), eviction may log an IN with Provisional.YES, although a
257    * checkpoint is not strictly active during this interval. See below for
258    * why this is safe.
259    *
260    * It is safe for eviction to log an IN as Provisional.YES for the last two
261    * special cases, because this does not cause incorrect recovery behavior. For
262    * recovery to work properly, it is only necessary that:
263    *
264    * + Provisional.NO is used for INs at the max flush level during an active
265    * checkpoint.
266    * + Provisional.YES or BEFORE_CKPT_END is used for INs below the max flush
267    * level, to avoid replaying an IN during recovery that may depend on a file
268    * deleted as the result of the checkpoint.
269    *
270    * You may ask why we don't use Provisional.YES for eviction when a checkpoint
271    * is not active. There are two reason, both related to performance:
272    *
273    * 1. This would be wasteful when an IN is evicted in between checkpoints, and
274    * that portion of the log is processed by recovery later, in the event of a
275    * crash. The evicted INs would be ignored by recovery, but the actions
276    * that caused them to be dirty would be replayed and the INs would be
277    * logged again redundantly.
278    * 2. Logging a IN provisionally will not count the old LSN as obsolete
279    * immediately, so cleaner utilization will be inaccurate until the a
280    * non-provisional parent is logged, typically by the next checkpoint. It
281    * is always important to keep the cleaner from stalling and spiking, to
282    * keep latency and throughput as level as possible.
283    *
284    * Therefore, it is safe to log with Provisional.YES in between checkpoints,
285    * but not desirable.
286    *
287    * Although we don't do this, it would be safe and optimal to evict with
288    * BEFORE_CKPT_END in between checkpoints, because it would be treated by
289    * recovery as if it were Provisional.NO. This is because the interval between
290    * checkpoints is only processed by recovery if it follows the last CkptEnd,
291    * and BEFORE_CKPT_END is treated as Provisional.NO if the IN follows the last
292    * CkptEnd.
293    *
294    * However, it would not be safe to evict an IN with BEFORE_CKPT_END during a
295    * checkpoint, when logging of the IN's ancestors does not occur according to
296    * the rules of the checkpoint. If this were done, then if the checkpoint
297    * completes and is used during a subsequent recovery, an obsolete offset for
298    * the old version of the IN will mistakenly be recorded. Below are two cases
299    * where BEFORE_CKPT_END is used correctly and one showing how it could be used
300    * incorrectly.
301    *
302    * 1. Correct use of BEFORE_CKPT_END when the checkpoint does not complete.
303    *
304    * 050 BIN-A
305    * 060 IN-B parent of BIN-A
306    * 100 CkptStart
307    * 200 BIN-A logged with BEFORE_CKPT_END
308    * 300 FileSummaryLN with obsolete offset for BIN-A at 050
309    * Crash and recover
310    *
311    * Recovery will process BIN-A at 200 (it will be considered
312    * non-provisional) because there is no following CkptEnd. It is
313    * therefore correct that BIN-A at 050 is obsolete.
314    *
315    * 2. Correct use of BEFORE_CKPT_END when the checkpoint does complete.
316    *
317    * 050 BIN-A
318    * 060 IN-B parent of BIN-A
319    * 100 CkptStart
320    * 200 BIN-A logged with BEFORE_CKPT_END
321    * 300 FileSummaryLN with obsolete offset for BIN-A at 050
322    * 400 IN-B parent of BIN-A, non-provisional
323    * 500 CkptEnd
324    * Crash and recover
325    *
326    * Recovery will not process BIN-A at 200 (it will be considered
327    * provisional) because there is a following CkptEnd, but it will
328    * process its parent IN-B at 400, and therefore the BIN-A at 200 will be
329    * active in the tree. It is therefore correct that BIN-A at 050 is
330    * obsolete.
331    *
332    * 3. Incorrect use of BEFORE_CKPT_END when the checkpoint does complete.
333    *
334    * 050 BIN-A
335    * 060 IN-B parent of BIN-A
336    * 100 CkptStart
337    * 200 BIN-A logged with BEFORE_CKPT_END
338    * 300 FileSummaryLN with obsolete offset for BIN-A at 050
339    * 400 CkptEnd
340    * Crash and recover
341    *
342    * Recovery will not process BIN-A at 200 (it will be considered
343    * provisional) because there is a following CkptEnd, but no parent
344    * IN-B is logged, and therefore the IN-B at 060 and BIN-A at 050 will be
345    * active in the tree. It is therefore incorrect that BIN-A at 050 is
346    * obsolete.
347    *
348    * This last case is what caused the LFNF in SR [#19422], when BEFORE_CKPT_END
349    * was mistakenly used for logging evicted BINs via CacheMode.EVICT_BIN.
350    * During the checkpoint, we evict BIN-A and log it with BEFORE_CKPT_END, yet
351    * neither it nor its parent are part of the checkpoint. After being counted
352    * obsolete, we crash and recover. Then the file containing the BIN (BIN-A at
353    * 050 above) is cleaned and deleted. During cleaning, it is not migrated
354    * because an obsolete offset was previously recorded. The LFNF occurs when
355    * trying to access this BIN during a user operation.
356    *
357    * CacheMode.EVICT_BIN
358    * -------------------
359    * Unlike in JE 4.0 where EVICT_BIN was first introduced, in JE 4.1 and later
360    * we do not use special rules when an IN is evicted. Since concurrent
361    * eviction and checkpointing are supported in JE 4.1, the above rules apply to
362    * EVICT_BIN as well as all other types of eviction.
363    */

603        */
604       private boolean isRunnable(CheckpointConfig config) {
605           /* Figure out if we're using log size or time to determine interval.*/
606           long useBytesInterval = 0;
607           long useTimeInterval = 0;
608           long nextLsn = DbLsn.NULL_LSN;

610           try {
611               if (config.getForce()) {
612                   runnable = true;
613                   return true;
614               }
615  
616               if (wakeupAfterNoWrites && needCheckpointForCleanedFiles()) {
617                   runnable = true;
618                   return true;
619               }
620  
621               if (config.getKBytes() != 0) {
622                   useBytesInterval = config.getKBytes() << 10;
623  
624               } else if (config.getMinutes() != 0) {
625                   /* Convert to millis. */
626                   useTimeInterval = config.getMinutes() * 60 * 1000;
627  
628               } else if (logSizeBytesInterval != 0) {
629                   useBytesInterval = logSizeBytesInterval;
630  
631               } else {
632                   useTimeInterval = timeInterval;
633               }
634  
635               /*
636                * If our checkpoint interval is defined by log size, check on how
637                * much log has grown since the last checkpoint.
638                */
639               if (useBytesInterval != 0) {
640                   nextLsn = envImpl.getFileManager().getNextLsn();
641  
642                   if (DbLsn.getNoCleaningDistance(
643                       nextLsn, lastCheckpointStart, logFileMax) >=
644                       useBytesInterval) {
645  
646                       runnable = true;
647                   }
648  
649               } else if (useTimeInterval != 0) {
650

652                    * Our checkpoint is determined by time. If enough time has
653                    * passed and some log data has been written, do a checkpoint.
654                    */
655                   final long lastUsedLsn =
656                       envImpl.getFileManager().getLastUsedLsn();
657  
658                   if (((System.currentTimeMillis() - lastCheckpointMillis) >=
659                        useTimeInterval) &&
660                       (DbLsn.compareTo(lastUsedLsn, lastCheckpointEnd) != 0)) {
661  
662                       runnable = true;
663                   }
664               }
665               return runnable;
666  
667           } finally {

669                   final StringBuilder sb = new StringBuilder();

676                       sb.append(" lastCkpt=");
677                       sb.append(DbLsn.getNoFormatString(lastCheckpointEnd));
678                   }
679                   sb.append(" time interval=").append(useTimeInterval);
680                   sb.append(" force=").append(config.getForce());
681                   sb.append(" runnable=").append(runnable);
682  
683                   LoggerUtils.finest(logger, envImpl, sb.toString());
684               }
685           }
686       }

712       /**

714        * thread when waking up, or it may be invoked programatically through the
715        * api.

717        * @param invokingSource a debug aid, to indicate who invoked this
718        * checkpoint. (i.e. recovery, the checkpointer daemon, the cleaner,
719        * programatically)
720        */
721       public synchronized void doCheckpoint(CheckpointConfig config,
722                                             String invokingSource) {
723           if (envImpl.isReadOnly()) {
724               return;
725           }

727           if (!isRunnable(config)) {
728               return;
729           }
730  
731           /*
732            * If minimizing recovery time is desired, then flush all the way to
733            * the top of the dbtree instead of stopping at the highest level last
734            * modified, so that only the root INs are processed by recovery.
735            */
736           final boolean flushAll = config.getMinimizeRecoveryTime();

738           /*
739            * If there are cleaned files to be deleted, flush an extra level to
740            * write out the parents of cleaned nodes. This ensures that no node
741            * will contain the LSN of a cleaned file.
742            *

744            * CLEANED and FULLY_PROCESSED states. For a FULLY_PROCESSED file, a
745            * pending LN may have been processed since the prior checkpoint.
746            * However, the BIN containing the LSN of the LN is guaranteed to be
747            * logged, so there is no need to increment maxFlushLevel. So we could
748            * optimize in the future and only set flushExtraLevel when some files
749            * are CLEANED (i.e., do not set flushExtraLevel when all files are
750            * FULLY_PROCESSED or cleanerState.isEmpty()).
751            */

753

755               cleaner.getFilesAtCheckpointStart();

757           final boolean flushExtraLevel = !cleanerState.isEmpty();

763           checkpointId++;

773            * coordinated.

780                   SingleItemEntry.create(

789                * firstActiveLsn is that all log entries for active transactions

791                * replaying LNs during recovery and will be stored in the CkptEnd
792                * entry.

798                * logging CkptStart. If it were instead retrieved before logging

998        * writes up the tree. If flushAll wasn't specified, we need only cascade

1002        * because we don't need to process lower INs during recovery because the
1003        * higher INs will end up pointing at them.
1004        */

1033                   while (true) {

1043                        * it in the dirty map, and prevent the DB from being
1044                        * deleted while we're working with it.
1045                        */

1047                           targetRef.dbId, -1 /*lockTimeout*/, dbCache);

1050

1053                               dirtyMap.getHighestFlushLevel(db);
1054  
1055                           if (currentLevelVal <= maxFlushLevel) { → [ALLOWCREATE, TRANSACTIONAL, READONLY]
1056

1058                                   db, targetRef, dirtyMap, maxFlushLevel,
1059                                   highPriority, fstats, true /*allowLogSubtree*/);

1071                        * get out of this loop, and re-throw the invalidating
1072                        * exception to indicate that the checkpoint did not
1073                        * succeed.
1074                        */
1075                       envImpl.checkIfInvalid();
1076                   }

1088            * already done by the checkpoint.

1096        * target, which means we flush the siblings of this target to promote
1097        * better cleaning throughput. The problem lies in the fact that

1099        * their parent is logged non-provisionally. On the other hand, we want to

1102        * files. (See [#16037].) These conflicting goals are reconciled by
1103        * flushing nodes in subtree grouping, because writing the non-provisional
1104        * parent of a set of provisionally written nodes frees the cleaner to work

1106        * tree consists of:
