181  /**
182    * Statistics for a single environment.
183    * <p>
184    * The statistics are logically grouped into several categories. Viewing the
185    * statistics through {@link EnvironmentStats#toString()} displays the values
186    * in these categories, as does viewing the stats through the {@link <a
187    * href="{@docRoot}/../jconsole/JConsole-plugin.html">JEMonitor mbean</a>}.
188    * Viewing the stats with {@link EnvironmentStats#toStringVerbose()} will
189    * provide more detailed descriptions of the stats and stat categories.
190    * <p>
191    * The current categories are:
192    * <ul>
193    * <li><b>IO</b>: Log file opens, reads, writes, cache misses.</li>
194    * <li><b>Cache</b>: Data, keys, internal btree nodes, locks and JE metadata
195    * reside in the cache. If the cache is too full, nodes, data and keys are
196    * periodically evicted to stay within the defined cache size.</li>
197    * <li><b>Log Cleaning</b>: Maintenance of JE's append only storage system.
198    * See this <a href="{@docRoot}/../GettingStartedGuide/logfilesrevealed.html">
199    * overview</a> of the logging system. Log files are removed as their
200    * contents become obsolete.</li>
201    * <li><b>Node Compression</b>: Internal btree nodes are compressed and removed
202    * when their contents become sparse. </li>
203    * <li><b>Checkpoints</b>: The frequency and extent of checkpointing activity.
204    * </li>
205    * <li><b>Environment</b>: Miscellaneous environment wide statistics.
206    * </li>
207    * <li><b>Locks</b>: Number of locking operations, contention on lock table.
208    * </li>
209    * </ul>
210    * @see <a href="{@docRoot}/../jconsole/JConsole-plugin.html">Viewing
211    * Statistics with JConsole</a>
212    */

659       /**
660        * Number of cleaner two-pass runs, which are a subset of the
661        * {@link #getNCleanerRuns() total cleaner runs}. The {@link
662        * #getCurrentMinUtilization() minimum} and {@link
663        * #getCurrentMaxUtilization() maximum} utilization values are used to
664        * drive cleaning.
665        * <p>
666        * The obsolete portion of the log includes data that has expired. An
667        * expiration histogram is stored for each file and used to compute the
668        * expired size. The minimum and maximum utilization are the lower and
669        * upper bounds of computed utilization. They are different only when the
670        * TTL feature is used, and some data in the file has expired while other
671        * data has become obsolete for other reasons, such as record updates,
672        * record deletions or checkpoints. In this case the strictly obsolete size
673        * and the expired size may overlap because they are maintained separately.
674        * <p>
675        * If they overlap completely then the minimum utilization is correct,
676        * while if there is no overlap then the maximum utilization is correct.
677        * Both utilization values trigger cleaning, but when there is significant
678        * overlap, the cleaner will perform two-pass cleaning.
679        * <p>
680        * In the first pass of two-pass cleaning, the file is read to recompute
681        * obsolete and expired sizes, but the file is not cleaned. As a result of
682        * recomputing the expired sizes, the strictly obsolete and expired sizes
683        * will no longer overlap, and the minimum and maximum utilization will be
684        * equal. If the file should still be cleaned, based on the recomputed
685        * utilization, it is cleaned as usual, and in this case the number of
686        * two-pass runs (this statistic) is incremented.
687        * <p>
688        * If the file should not be cleaned because its recomputed utilization is
689        * higher than expected, the file will not be cleaned. Instead, its
690        * recomputed expiration histogram, which has size information that now
691        * does not overlap with the strictly obsolete data, is stored for future
692        * use. By storing the revised histogram, the cleaner can select the most
693        * appropriate files for cleaning in the future. In this case the number of
694        * {@link #getNCleanerRevisalRuns() revisal runs} is incremented, and the
695        * number of {@link #getNCleanerRuns() total runs} is not incremented.
696        *
