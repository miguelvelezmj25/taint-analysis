56  /**
57    * With this class objects can be serialized to XML instead into a binary
58    * format. It uses introspection (cf. beans) to retrieve the data from the given
59    * object, i.e. it can only access beans-conform fields automatically.
60    * <p>
61    * The generic approach of writing data as XML can be overriden by adding custom
62    * methods for reading/writing in a derived class (cf. <code>m_Properties</code>, <code>m_CustomMethods</code>).<br>
63    * Custom read and write methods must have the same signature (and also be
64    * <code>public</code>!) as the <code>readFromXML</code> and
65    * <code>writeToXML</code> methods. Methods that apply to the naming rule
66    * <code>read + property name</code> are added automatically to the list of
67    * methods by the method
68    * <code>XMLSerializationMethodHandler.addMethods(...)</code>.
69    * <p>
70    * Other properties that are not conform the bean set/get-methods have to be
71    * processed manually in a derived class (cf.
72    * <code>readPostProcess(Object)</code>, <code>writePostProcess(Object)</code>).
73    * <p>
74    * For a complete XML serialization/deserialization have a look at the
75    * <code>KOML</code> class.
76    * <p>
77    * If a stored class has a constructor that takes a String to initialize (e.g.
78    * String or Double) then the content of the tag will used for the constructor,
79    * e.g. from
80    *
81    * <pre>
82    * <object name="name" class="String" primitive="no">Smith</object>
83    * </pre>
84    *
85    * "Smith" will be used to instantiate a String object as constructor argument.
86    * <p>
87    *
88    * @see KOML
89    * @see #fromXML(Document)
90    * @see #toXML(Object)
91    * @see #m_Properties
92    * @see #m_CustomMethods
93    * @see #readPostProcess(Object)
94    * @see #writePostProcess(Object)
95    * @see #readFromXML(Element)
96    * @see #writeToXML(Element, Object, String)
97    *
98    * @author FracPete (fracpete at waikato dot ac dot nz)
