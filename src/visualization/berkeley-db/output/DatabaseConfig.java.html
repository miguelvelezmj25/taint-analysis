466       /**
467        * By default, a byte by byte lexicographic comparison is used for btree
468        * keys. To customize the comparison, supply a different Comparator.
469        *
470        * <p>Note that there are two ways to set the comparator: by specifying the
471        * class or by specifying a serializable object. This method is used to
472        * specify a serializable object. The comparator class must implement
473        * java.util.Comparator and must be serializable. JE will serialize the
474        * Comparator and deserialize it when subsequently opening the
475        * database.</p>
476        *
477        * <p>If a comparator needs to be initialized before it is used or needs
478        * access to the environment's ClassLoader property, it may implement the
479        * {@link DatabaseComparator} interface.</p>
480        *
481        * <p>The Comparator.compare() method is passed the byte arrays that are
482        * stored in the database. If you know how your data is organized in the
483        * byte array, then you can write a comparison routine that directly
484        * examines the contents of the arrays. Otherwise, you have to reconstruct
485        * your original objects, and then perform the comparison. See the <a
486        * href="{@docRoot}/../GettingStartedGuide/comparator.html"
487        * target="_top">Getting Started Guide</a> for examples.</p>
488        *
489        * <p><em>WARNING:</em> There are several special considerations that must
490        * be taken into account when implementing a comparator.<p>
491        * <ul>
492        * <li>Comparator instances are shared by multiple threads and comparator
493        * methods are called without any special synchronization. Therefore,
494        * comparators must be thread safe. In general no shared state should be
495        * used and any caching of computed values must be done with proper
496        * synchronization.</li>
497        *
498        * <li>Because records are stored in the order determined by the
499        * Comparator, the Comparator's behavior must not change over time and
500        * therefore should not be dependent on any state that may change over
501        * time. In addition, although it is possible to change the comparator
502        * for an existing database, care must be taken that the new comparator
503        * provides compatible results with the previous comparator, or database
504        * corruption will occur.</li>
505        *
506        * <li>JE uses comparators internally in a wide variety of circumstances,
507        * so custom comparators must be sure to return valid values for any two
508        * arbitrary keys. The user must not make any assumptions about the
509        * range of key values that might be compared. For example, it's possible
510        * for the comparator may be used against previously deleted values.</li>
511        * </ul>
512        *
513        * <p>A special type of comparator is a <em>partial comparator</em>, which
514        * allows for the keys of a database to be updated, but only if the updates
515        * do not change the relative order of the keys. For example, if a database
516        * uses strings as keys and a case-insensitive comparator, it is possible
517        * to change the case of characters in the keys, as this will not change
518        * the ordering of the keys. Another example is when the keys contain
519        * multiple fields but uniquely identify each record with a single field.
520        * The partial comparator could then compare only the single identifying
521        * field, allowing the rest of the fields to be updated. A query
522        * ({@link Cursor#getSearchKey Cursor.getSearchKey}, for example) could
523        * then be performed by passing a partial key that contains only the
524        * identifying field.
525        *
526        * <p><strong>WARNING:</strong> To allow for key updates in situations
527        * like those described above, all partial comparators must implement the
528        * {@link PartialComparator} tag interface. Otherwise, BDB JE will raise
529        * an exception if an attempt is made to update a key in a database whose
530        * comparators do not implement PartialComparator. See "Upgrading from JE
531        * 5.0 or earlier" in the change log and the {@link PartialComparator}
532        * javadoc for more information.</p>
533        * <p>
534        * Another special type of comparator is a <em>binary equality</em>
535        * comparator, which considers two keys to be equal if and only if they
536        * have the same length and they are equal byte-per-byte. All binary
537        * equality comparators must implement the {@link BinaryEqualityComparator}
538        * interface. The significance of binary equality comparators is that they
539        * make possible certain internal optimizations, like the "blind puts"
540        * optimization, described in
541        * {@link BinaryEqualityComparator}
542        * <p>
543        * The comparator for an existing database will not be overridden unless
544        * setOverrideBtreeComparator() is set to true.
545        *
546        * @return this
547        */
